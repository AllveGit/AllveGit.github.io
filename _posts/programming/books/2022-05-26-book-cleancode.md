# 🚀 클린 코드 (Clean Code)

> 개발자들은 항상 좋은 코드를 짜기 위해 고민하고 연구할 것입니다.

클린 코드, 개발자라면 한번씩 들어봤을 책입니다.  
흔히 개발자들의 필독서라고 불리우는 책입니다.  
  
좋은 코드란 무엇인가? 에 대해 길잡이 역할을 해주며, <u>가독성이 좋고 유지보수가 어렵지 않은 코드</u>를 짜기 위해 어떤 방법들을 사용할 수 있는지 알려주는 책입니다.

## 🔥 깨끗한 코드란?  
  
> 코드에는 좋은 코드와 나쁜 코드가 있다.  
  
나쁜 코드는 회사를 망하게 할수도 있는 큰 악영향을 끼칩니다.  
예를 들어, 어떤 이슈 하나를 수정하면 수정한 코드에 따라 또 여러 사이드 이펙트가 터지는 등...  
프로그래머라면 무조건 나쁜 코드를 경험해봤을 것입니다.  
  
<u>나쁜 코드가 쌓이게 되면, 그만큼 생산성이 떨어지게 됩니다.</u>  
계속 방치한 채로 작업을 진행한다면, 나중에는 새로 만드는 것이 더 빠를 수준의 생산성이 되버리고 말 것입니다.  
<u>그렇기에 오히려 나쁜 코드가 단기간적으로는 작업이 빠를 수 있더라도, 장기간적으로 보았을 때는 결국 작업 시간이 더 많이 들게 됩니다.</u>

이런 나쁜 코드를 만드는 악습관중 하나는, 일정에 맞추느라 기능만 급히 만들고 나중에 정리해야지하고 미루는 습관입니다.  
일정에 맞추어 좋은 코드를 사수하는 것이 훌륭한 프로그래머의 책임이라고 로버트 C.마틴은 말합니다.  
  
그러면 좋은 코드, 즉 깨끗한 코드는 무엇일까요? 책을 읽고 이해한다면 그 해답에 조금은 가까워질 수 있다고 말합니다.  
  
❗  
로버트 C.마틴은 자신이 생각하고 있는 방법이 최선이라고 믿고 알려주고 있습니다.  
하지만, 로버트 C.마틴도 자신의 생각이 절대로 옳다 라는 단정은 하지 말아달라고 합니다.  
그러니, 자신의 생각하는 깨끗한 코드가 무엇인지 알려주는 책이며, 이 책이 완벽한 정답이라고는 할 수 없습니다.

# 🚀 작명
  
> 설문조사 결과, 프로그래머가 가장 시간을 많이 쏟는 작업은 "작명" 이라고 합니다.  
  
그만큼, 이름은 코드에서 아주 중요한 역할을 맡으며 내가 작성한 코드의 의미를 전달하기 위해 필요한 작업입니다.  
아무 생각없이 작명하는 것은 나쁜 코드로 가는 지름길을 넘어 고속도로 수준일 것입니다.  
  
어떻게 이름을 작명해야 좋은 코드를 작성할 수 있는지 이 책은 설명합니다.  
  
## 🔥 의도를 분명히 밝혀라  
  
```java
int y;
int m;
int d;
int s;
```
  
위 예제코드의 변수 이름들에는 아무 의미를 파악할 수 없습니다.  
  
```java
int year;
int month;
int day;
int second;
```  
  
이런 식으로 의미 있는 변수 이름으로 작명하면 날짜를 나타낼려고 하는 것을 부가 설명없이 알 수 있습니다.   
  
이는, 함수에서도 마찬가지입니다.  
  
```java
public List<int[]> getLists() {
    List<int[]> list1 = new ArrayList<int[]>();
    for (int[] x : TheList)
        if (x[0] == 4)
            list1.add(x);

    return list1;
}
```

이 예제코드는 한번에 이해가 가지 않습니다. 왜냐하면 생각없이 지은 이름이 함수의 의도를 불분명하게 만들었기 때문입니다.  
  
```java
public List<Cell> getFlaggedCells() {
    List<Cell> flaggedCells = new ArrayList<Cell>();
    for (Cell cell : gameBoard)
        if (cell.isFlagged())
            flaggedCells.add(cell);

    return flaggedCells;
}
```

코드를 개선하면 어떤 플래그가 켜져있는 셀들을 추출하는 함수인것을 한 눈에 알아볼 수 있습니다.  
이만큼, <u>이름이 가져다주는 코드의 효과는 매우 큽니다.</u>  
  
## 🔥 오해를 살만한 정보를 피하라  

그룹을 짓는 콜렉션 변수의 이름 마지막에 List를 자주 넣습니다. (ex. userList)  
하지만, 프로그래머에게 List는 특별한 의미입니다. List 자료형이 아닌데 List라 작명하면 혼동을 줄 수 있습니다.  
이는, users, userGroup등으로 이름을 바꿀 수 있습니다.  

소문자 L은 숫자 1처럼 보일 수 있고, 대문자 O는 숫자 0처럼 보입니다.  
<u>작명을 할 때 헷갈릴만한 문자는 사용하지 않는 것이 좋습니다.</u>  
  
## 🔥 의미 있게 구분하라  
  
연속된 숫자를 덧붙이거나 불용어를 추가하는 방식은 의미있게 작명을 하는 방식이 아닙니다.

```java
class Product {...};
class ProductInfo {...};
class ProductData {...};
```  
  
예를 들어, 이 세가지의 클래스를 정의했다고 했을 때 차이점을 정확히 파악하기는 힘듭니다.
Info, Data같은 불용어를 붙였다고 해서 차이점을 구분하기는 커녕 오히려 모호해지는 현상이 발생하고 맙니다.  
  
그렇기 때문에, <u>불용어를 사용한 이름은 중복으로 봐도 무방합니다.</u> 읽는 사람이 차이를 구분할 수 있도록 이름을 작명해야 합니다.  
  
## 🔥 발음하기 쉬운 이름을 사용하라  
  
이름을 지을 때 축약을 하여 발음조차 힘든 이름이 되었다면, 이 이름을 부를 때 난감한 상황이 발생합니다.  
특히, 중요한 로직을 담당하는 곳의 이름이면 수도 없이 개발자들에게 불릴 이름입니다.  
우리가 현실에서 사용하는 언어로 발음하기 쉬운 이름을 채택하도록 합시다.  
  
## 🔥 검색하기 쉬운 이름을 사용하라  
  
요즘의 IDE에서 변수나 함수이름을 검색할 수 있다.  
그렇기에, IDE에서 검색할 때 쉬운 이름을 사용해야 나중에 그 변수나 함수가 참조한 곳들을 수정하려면 검색하기 쉬워야한다.
    
예를 들어, 찾아야할 값이 상수인 5라면, 5라는 값이 쓰이는 수식, 변수 이름 및 함수등 정말 많은 곳이 검색이 될 것입니다.  
이 많은 곳중에서 내가 찾고 싶은 부분을 찾는 것은 굉장히 어려운 일일 것입니다.  
  
그렇기에 이 상수 5를 REWARD_RECEIVE_NUM 같은 이름으로 바꾼다면 확실히 찾을 수 있게 될 것입니다.  
이름이 짧아도 그 이름이 포함되는 곳이 전부 검색될 것이므로, <u>검색이 중요할 것 같은 것들의 이름은 길게 짓도록 하는 것이 좋습니다.</u>  
  
## 🔥 헝가리 표기법, 멤버 변수 접두어를 사용하지 마라  
  
```java
int iNum;
Bool bResult;
String strLine;
```  
  
이렇게 변수 및 함수 인자의 이름 앞에 데이터 타입을 명시해주는 것이 헝가리 표기법입니다.  
요즘의 IDE는 변수 및 함수 인자의 타입이 어떤 타입인지 알려주기에 굳이 이름에 명시해줄 필요가 없습니다.  
  
게다가, 데이터 타입을 명시하면 데이터 타입이 바뀌었을 때 이름까지 다 바꿔주어야 하는 수고가 들며,  
헝가리 표기법을 쓰면 쓸데없는 접두어가 붙어 코드의 가독성이 매우 떨어지게 됩니다.  
  
이렇게 단점이 많은 <u>헝가리 표기법은 사용하지 않는 것이 좋습니다.</u>  
  
```java
public class Test {
  private String m_Description;
  private int m_Count;
}
```
  
사람들은 접두어 또는 접미어를 무시하고 이름을 해독하는 방식을 재빨리 익힙니다.  
그렇기에 접두어 또는 접미어는 코드를 읽을 때 관심밖으로 밀려날 수 밖에 없습니다.  
  
게다가, 최신의 IDE들은 멤버변수의 색상을 다르게 표현해주므로 굳이 접두어를 붙이지 않더라도 멤버변수인 것을 알 수 있습니다.  
  
그렇기에 <u>멤버변수에는 접두어를 붙여 멤버변수인 것을 나타내지 않는 것이 좋습니다.</u>  
  
## 🔥 클래스, 메서드 이름  

> 클래스와 객체 이름은 명사나 명사구가 적합합니다.  

예를 들어, Customer, WikiPage, Account, AddressParser등이 좋은 이름입니다.  
의미가 모호한 Manager, Processor, Data, Info 등과 같은 단어는 피하는 것이 좋습니다.  

> 메서드 이름은 동사나 동사구가 적합합니다.  
  
예를 들어, PostPayment(), DeletePage(), SavePage()등이 좋은 이름입니다.  
접근자는 Get, 변경자는 Set, 조건자는 Is를 접두어로 사용하는 것이 좋습니다.  
ex) GetName(), SetName(), IsName()  
  
생성자를 중복 정의하여 오버로딩하는 상황이 있다면 생성자 오버로딩 방식이 아닌, <u>정적 팩토리 메서드를 사용하는 것이 좋습니다.</u>  
  
```java
Complex badFulcrumPoint = new Complex(23.0);                // 1
Complex goodFulcrumPoint = Complex.FromRealNumber(23.0);    // 2
```  
  
즉, 1번 방식보다 2번 방식이 좋다는 의미입니다.  
왜냐하면 1번 방식은 인자로 어떤 값을 넣어줘야 하는지 한번에 모를 뿐더러 저런 생성자가 있다는 것도 알기 힘듭니다.  
2번 방식은 어떤 인자로 객체로 생성되는지 이름만으로 알 수 있기에 더욱 가독성이 좋아집니다.  
  
## 🔥 한 개념에 한 단어를 사용하라    
  
비슷한 기능을 하는 함수들이 클래스마다 get, fetch, retrieve로 제각각 부르면 혼란이 일어나게 됩니다.  
마찬가지로, 비슷한 기능을 하는 클래스들이 controller, manager, driver를 섞어 쓰게 되어도 마찬가지일 것입니다.  
  
<u>하나의 추상화된 개념에는 동일한 단어를 사용해야 코드에 혼란이 찾아오지 않고 일관성이 생기게 됩니다.</u>  
  
또 하나의 예로, 값을 + 수식으로 더하는 Add와 리스트에 값을 추가하는 Add는 맥락이 다릅니다.  
이는 똑같은 개념이 아니지만 동시에 Add를 쓰면 사용하는 입장에서 헷갈리게 될 수 있습니다.  
이럴 때는 똑같은 단어를 사용하는 것이 아닌 insert, append등으로 대체하여 사용해야 합니다.
