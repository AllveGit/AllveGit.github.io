# 🚀 클린 코드 (Clean Code)

> 개발자들은 항상 좋은 코드를 짜기 위해 고민하고 연구할 것입니다.

클린 코드, 개발자라면 한번씩 들어봤을 책입니다.  
흔히 개발자들의 필독서라고 불리우는 책입니다.  
  
좋은 코드란 무엇인가? 에 대해 길잡이 역할을 해주며, <u>가독성이 좋고 유지보수가 어렵지 않은 코드</u>를 짜기 위해 어떤 방법들을 사용할 수 있는지 알려주는 책입니다.

## 🔥 깨끗한 코드란?  
  
> 코드에는 좋은 코드와 나쁜 코드가 있다.  
  
나쁜 코드는 회사를 망하게 할수도 있는 큰 악영향을 끼칩니다.  
예를 들어, 어떤 이슈 하나를 수정하면 수정한 코드에 따라 또 여러 사이드 이펙트가 터지는 등...  
프로그래머라면 무조건 나쁜 코드를 경험해봤을 것입니다.  
  
<u>나쁜 코드가 쌓이게 되면, 그만큼 생산성이 떨어지게 됩니다.</u>  
계속 방치한 채로 작업을 진행한다면, 나중에는 새로 만드는 것이 더 빠를 수준의 생산성이 되버리고 말 것입니다.  
<u>그렇기에 오히려 나쁜 코드가 단기간적으로는 작업이 빠를 수 있더라도, 장기간적으로 보았을 때는 결국 작업 시간이 더 많이 들게 됩니다.</u>

이런 나쁜 코드를 만드는 악습관중 하나는, 일정에 맞추느라 기능만 급히 만들고 나중에 정리해야지하고 미루는 습관입니다.  
일정에 맞추어 좋은 코드를 사수하는 것이 훌륭한 프로그래머의 책임이라고 로버트 C.마틴은 말합니다.  
  
그러면 좋은 코드, 즉 깨끗한 코드는 무엇일까요? 책을 읽고 이해한다면 그 해답에 조금은 가까워질 수 있다고 말합니다.  
  
❗  
로버트 C.마틴은 자신이 생각하고 있는 방법이 최선이라고 믿고 알려주고 있습니다.  
하지만, 로버트 C.마틴도 자신의 생각이 절대로 옳다 라는 단정은 하지 말아달라고 합니다.  
그러니, 자신의 생각하는 깨끗한 코드가 무엇인지 알려주는 책이며, 이 책이 완벽한 정답이라고는 할 수 없습니다.
{: .notice--warning}  
  
# 🚀 작명
  
> 설문조사 결과, 프로그래머가 가장 시간을 많이 쏟는 작업은 "작명" 이라고 합니다.  
  
그만큼, 이름은 코드에서 아주 중요한 역할을 맡으며 내가 작성한 코드의 의미를 전달하기 위해 필요한 작업입니다.  
아무 생각없이 작명하는 것은 나쁜 코드로 가는 지름길을 넘어 고속도로 수준일 것입니다.  
  
어떻게 이름을 작명해야 좋은 코드를 작성할 수 있는지 이 책은 설명합니다.  
  
## 🔥 의도를 분명히 밝혀라  
  
```java
int y;
int m;
int d;
int s;
```
  
위 예제코드의 변수 이름들에는 아무 의미를 파악할 수 없습니다.  
  
```java
int year;
int month;
int day;
int second;
```  
  
이런 식으로 의미 있는 변수 이름으로 작명하면 날짜를 나타낼려고 하는 것을 부가 설명없이 알 수 있습니다.   
  
이는, 함수에서도 마찬가지입니다.  
  
```java
public List<int[]> getLists() {
    List<int[]> list1 = new ArrayList<int[]>();
    for (int[] x : TheList)
        if (x[0] == 4)
            list1.add(x);

    return list1;
}
```

이 예제코드는 한번에 이해가 가지 않습니다. 왜냐하면 생각없이 지은 이름이 함수의 의도를 불분명하게 만들었기 때문입니다.  
  
```java
public List<Cell> getFlaggedCells() {
    List<Cell> flaggedCells = new ArrayList<Cell>();
    for (Cell cell : gameBoard)
        if (cell.isFlagged())
            flaggedCells.add(cell);

    return flaggedCells;
}
```

코드를 개선하면 어떤 플래그가 켜져있는 셀들을 추출하는 함수인것을 한 눈에 알아볼 수 있습니다.  
이만큼, <u>이름이 가져다주는 코드의 효과는 매우 큽니다.</u>  
  
## 🔥 오해를 살만한 정보를 피하라  

그룹을 짓는 콜렉션 변수의 이름 마지막에 List를 자주 넣습니다. (ex. userList)  
하지만, 프로그래머에게 List는 특별한 의미입니다. List 자료형이 아닌데 List라 작명하면 혼동을 줄 수 있습니다.  
이는, users, userGroup등으로 이름을 바꿀 수 있습니다.  

소문자 L은 숫자 1처럼 보일 수 있고, 대문자 O는 숫자 0처럼 보입니다.  
<u>작명을 할 때 헷갈릴만한 문자는 사용하지 않는 것이 좋습니다.</u>  
  
## 🔥 의미 있게 구분하라  
  
연속된 숫자를 덧붙이거나 불용어를 추가하는 방식은 의미있게 작명을 하는 방식이 아닙니다.

```java
class Product {...};
class ProductInfo {...};
class ProductData {...};
```  
  
예를 들어, 이 세가지의 클래스를 정의했다고 했을 때 차이점을 정확히 파악하기는 힘듭니다.
Info, Data같은 불용어를 붙였다고 해서 차이점을 구분하기는 커녕 오히려 모호해지는 현상이 발생하고 맙니다.  
  
그렇기 때문에, <u>불용어를 사용한 이름은 중복으로 봐도 무방합니다.</u> 읽는 사람이 차이를 구분할 수 있도록 이름을 작명해야 합니다.  
  
## 🔥 발음하기 쉬운 이름을 사용하라  
  
이름을 지을 때 축약을 하여 발음조차 힘든 이름이 되었다면, 이 이름을 부를 때 난감한 상황이 발생합니다.  
특히, 중요한 로직을 담당하는 곳의 이름이면 수도 없이 개발자들에게 불릴 이름입니다.  
우리가 현실에서 사용하는 언어로 발음하기 쉬운 이름을 채택하도록 합시다.  
  
## 🔥 검색하기 쉬운 이름을 사용하라  
  
요즘의 IDE에서 변수나 함수이름을 검색할 수 있다.  
그렇기에, IDE에서 검색할 때 쉬운 이름을 사용해야 나중에 그 변수나 함수가 참조한 곳들을 수정하려면 검색하기 쉬워야한다.
    
예를 들어, 찾아야할 값이 상수인 5라면, 5라는 값이 쓰이는 수식, 변수 이름 및 함수등 정말 많은 곳이 검색이 될 것입니다.  
이 많은 곳중에서 내가 찾고 싶은 부분을 찾는 것은 굉장히 어려운 일일 것입니다.  
  
그렇기에 이 상수 5를 REWARD_RECEIVE_NUM 같은 이름으로 바꾼다면 확실히 찾을 수 있게 될 것입니다.  
이름이 짧아도 그 이름이 포함되는 곳이 전부 검색될 것이므로, <u>검색이 중요할 것 같은 것들의 이름은 길게 짓도록 하는 것이 좋습니다.</u>  
  
## 🔥 헝가리 표기법, 멤버 변수 접두어를 사용하지 마라  
  
```java
int iNum;
Bool bResult;
String strLine;
```  
  
이렇게 변수 및 함수 인자의 이름 앞에 데이터 타입을 명시해주는 것이 헝가리 표기법입니다.  
요즘의 IDE는 변수 및 함수 인자의 타입이 어떤 타입인지 알려주기에 굳이 이름에 명시해줄 필요가 없습니다.  
  
게다가, 데이터 타입을 명시하면 데이터 타입이 바뀌었을 때 이름까지 다 바꿔주어야 하는 수고가 들며,  
헝가리 표기법을 쓰면 쓸데없는 접두어가 붙어 코드의 가독성이 매우 떨어지게 됩니다.  
  
이렇게 단점이 많은 <u>헝가리 표기법은 사용하지 않는 것이 좋습니다.</u>  
  
```java
public class Test {
  private String m_Description;
  private int m_Count;
}
```
  
사람들은 접두어 또는 접미어를 무시하고 이름을 해독하는 방식을 재빨리 익힙니다.  
그렇기에 접두어 또는 접미어는 코드를 읽을 때 관심밖으로 밀려날 수 밖에 없습니다.  
  
게다가, 최신의 IDE들은 멤버변수의 색상을 다르게 표현해주므로 굳이 접두어를 붙이지 않더라도 멤버변수인 것을 알 수 있습니다.  
  
그렇기에 <u>멤버변수에는 접두어를 붙여 멤버변수인 것을 나타내지 않는 것이 좋습니다.</u>  
  
## 🔥 클래스, 메서드 이름  

> 클래스와 객체 이름은 명사나 명사구가 적합합니다.  

예를 들어, Customer, WikiPage, Account, AddressParser등이 좋은 이름입니다.  
의미가 모호한 Manager, Processor, Data, Info 등과 같은 단어는 피하는 것이 좋습니다.  

> 메서드 이름은 동사나 동사구가 적합합니다.  
  
예를 들어, PostPayment(), DeletePage(), SavePage()등이 좋은 이름입니다.  
접근자는 Get, 변경자는 Set, 조건자는 Is를 접두어로 사용하는 것이 좋습니다.  
ex) GetName(), SetName(), IsName()  
  
생성자를 중복 정의하여 오버로딩하는 상황이 있다면 생성자 오버로딩 방식이 아닌, <u>정적 팩토리 메서드를 사용하는 것이 좋습니다.</u>  
  
```java
Complex badFulcrumPoint = new Complex(23.0);                // 1
Complex goodFulcrumPoint = Complex.FromRealNumber(23.0);    // 2
```  
  
즉, 1번 방식보다 2번 방식이 좋다는 의미입니다.  
왜냐하면 1번 방식은 인자로 어떤 값을 넣어줘야 하는지 한번에 모를 뿐더러 저런 생성자가 있다는 것도 알기 힘듭니다.  
2번 방식은 어떤 인자로 객체로 생성되는지 이름만으로 알 수 있기에 더욱 가독성이 좋아집니다.  
  
## 🔥 한 개념에 한 단어를 사용하라    
  
비슷한 기능을 하는 함수들이 클래스마다 get, fetch, retrieve로 제각각 부르면 혼란이 일어나게 됩니다.  
마찬가지로, 비슷한 기능을 하는 클래스들이 controller, manager, driver를 섞어 쓰게 되어도 마찬가지일 것입니다.  
  
<u>하나의 추상화된 개념에는 동일한 단어를 사용해야 코드에 혼란이 찾아오지 않고 일관성이 생기게 됩니다.</u>  
  
또 하나의 예로, 값을 + 수식으로 더하는 Add와 리스트에 값을 추가하는 Add는 맥락이 다릅니다.  
이는 똑같은 개념이 아니지만 동시에 Add를 쓰면 사용하는 입장에서 헷갈리게 될 수 있습니다.  
이럴 때는 똑같은 단어를 사용하는 것이 아닌 insert, append등으로 대체하여 사용해야 합니다.  

# 🚀 함수

> 코딩을 하면서 가장 많이 접하게 되는 것은 함수일 것입니다. 그만큼 코드의 기본 단위라고 할 수 있습니다.
  
함수를 잘 짜는 법을 알 수 있다면 클린 코드에 가까워질 수 있다고 장담할 수 있습니다.  
  
함수는 기본적으로 중복 코드를 없애기 위해 만들어졌다고 생각합니다.  
<u>그렇기에, 함수를 사용할 때 중복된 코드들을 없애는 것을 1차 목표로 생각해야 합니다.</u>  
  
## 🔥 작게 만들어라  
  
<u>함수는 긴 것보다는 작은 것이 훨씬 좋습니다.</u>  
함수가 길다면, 그 함수는 이해하기 어려울 확률이 매우 높고, 함수는 한 가지 일만 해야된다는 SRP 원칙도 위배할 확률이 높습니다.  
그렇기에 긴 함수보다는 작은 함수가 훨씬 낫습니다.  
  
들여쓰기 수준이 1단 ~ 2단으로 유지되는 것이 가독성에 좋습니다.
  
## 🔥 한 가지만 해라 (SRP)  

```java
public static String renderPlayerInfo(Creature inCreature) {
  boolean isPlayer = inCreature.hasAttribute("Player");
  boolean isExistCreature = Creatures.Find(inCreature.id);

  if (isPlayer && isExistCreature)
  {
    if (inCreature.getHP() > 0)
    {
      inCreature.Kill();
      System.out.println("Player Dead!");
    }
    else
    {
      System.out.println("Player Alive!");
    }

    ....
  }
  else
  {
    return "";
  }
}
```  
  
위 함수는 총체적 난국의 함수입니다. 함수 이름은 renderPlayerInfo입니다.  
함수 이름만 보면 플레이어의 정보를 출력해주는 함수이라고 판단할 수 있습니다.  
하지만, 실제로는 이 크리처가 플레이어인지 확인, 이 크리처가 존재하는 크리처인지 확인, 크리처의 HP가 0이하면 죽는 함수까지 호출합니다.  
함수 이름에 맞지 않는 행동들까지 하며, 심지어는 출력 함수에서 크리처의 상태까지 바꿔버립니다. 
  
이 함수는 이렇게 개선할 수 있습니다.

```java
public static String renderPlayerInfo(Creature inCreature) {
  if (!isPlayer(inCreature) || !isExist(inCreature))
  {
    return "";
  }

  if (inCreature.isDead())
  {
    System.out.println("Player Dead!");
  }
  else
  {
    System.out.println("Player Alive!");
  }

  ....
}
```  
  
한눈에 봐도 간단해졌으며, 가독성이 좋아진 것이 느껴집니다.  
함수 이름과 다른 행동들이 함수로 빠졌으므로 이 함수는 플레이어의 정보를 출력해주는 한 가지의 일만 하는 함수로 바뀌었습니다.  
  
## 🔥 switch문  
  
switch문은 작게 만들기 힘듭니다.  
특히 <u>한 가지의 작업만 하는</u> switch문은 더더욱 힘듭니다.  
  
그래서 보통 switch문은 생성 Factory를 만들 때 사용합니다.  
  
```java
public abstract class Employee {
  public abstract boolean isPayDay();
  public abstract Money calculatePay();
  public abstract void deliverPay(Money pay);
}

public interface EmployeeFactory {
  public Employee makeEmployee(EmployeeRecord r) throws InvalidEmployeeType;
}

public class EmployeeFactoryImpl {
  public Employee makeEmployee(EmployeeRecord r) throws InvalidEmployeeType {
    switch (r.type) {
      case COMMISSIONED:
        return new CommissionEmployee(r);
      case HOURLY:
        return new HourlyEmployee(r);
      case SALARIED:
        return new SalariedEmployee(r);
      default:
        throw new InvalidEmployeeType(r.type);
    }
  }
}
```  
  
팩토리를 이용하면 새로운 객체타입을 추가할 때 빼고는 switch문을 수정할 필요가 없어집니다.  
팩토리 메서드만 호출하면 되므로 switch문이 외부로 노출될 일이 없어 깔끔한 코드를 만들 수 있습니다.

## 🔥 서술적인 이름을 사용하라  
  
> 코드는 신문과도 같습니다.  
  
이 뜻은 코드를 읽을 때 신문처럼 위에서 아래로, 서술적으로 읽는다는 의미입니다.  
그렇다면 <u>가독성이 좋은 코드는 서술적으로 읽히는 것도 포함될 것입니다.</u>  
  
함수 이름을 서술적인 이름으로 짓는다면, 함수가 어떤 역할을 하는지 코드를 보지도 않고 대략적으로 알 수 있습니다.  
함수 이름으로 어떤 역할을 하는지 알고 함수 내부 코드를 본다면 분석 속도가 훨씬 빨라질 것입니다.  
  
<u>그렇기 때문에, 함수 이름을 작명할 때에는 많은 시간을 들일 가치가 있습니다.</u>

## 🔥 함수 인수  
  
함수에는 매개변수, 인자, 인수라 불리우는 변수들을 전달할 수 있습니다.  
  
코드를 보다보면 가로 스크롤을 해야할 정도로 인수들이 엄청 많은 긴 함수를 본 적이 있을 겁니다.  
그리고, 이 코드들의 가독성이 매우 현저하게 낮은 것도 깨달을 수 있습니다.  
  
<u>즉, 함수 인수도 가급적이면 작게 만드는 것을 생각해야 합니다.</u>  
  
클린 코드에서는 함수 인수가 웬만하면 0 ~ 3개여야 한다고 말합니다.  
<u>함수 인수들이 이 개수를 넘어가면 인수들을 구조체나 클래스로 만들어 전달하는 방식을 사용하는 것이 좋습니다.</u>  
  
함수 인수중, 플래그 인수는 굉장히 좋지 않은 인수입니다. 플래그만으로 객체가 어떤 작동을 하는지 알 수 없기 때문입니다.  
  
```java
render(true); 
```  
  
여기서 인수로 넘기는 true, false를 넘겼을 때 어떤 작동을 할지 한눈에 봐도 모를수밖에 없습니다.  
그렇기에 플래그 인수는 가급적 사용하지 않는 것이 좋습니다.    

## 🔥 부수 효과를 일으키지 마라  
  
앞에서 함수는 한 가지 일만 해야한다는 원칙(SRP)이 있었습니다.  
  
하지만, 함수들중에서 예상치 못하게 클래스 변수를 바꿔버리거나, 전역 변수를 수정한다는 등 부수효과들이 있는 함수들이 있습니다.  
이런 함수들은 시간적 결합 혹은 실행 순서에 의존하게 되어버립니다.  
  
```java
public class PlayerInfo
{
  private int hp;
  private boolean isAttack;

  public boolean CheckAlive() {
    if (hp > 0)
    {
      return true;
    }
    else
    {
      isAttack = false;
      return false;
    }
  }  
}
```  
  
위 예제 코드만 보더라도, 플레이어가 살아있는지 확인만하는 함수에서 죽었다면 공격중이라는 플래그 변수를 false로 만듭니다.  
이는 후에 실행할 함수들에서 잘못된 실행결과를 도출시키게 될 겁니다.  
  
<u>부수효과를 일으키는 함수들이 존재한다면 그것은 클린코드를 점점 오염시키는 주범이 될 수 있습니다.</u>

## 🔥 오류코드보다 예외를 사용하라  
  
오류 처리를 할 때 나만의 오류코드를 정의하여 처리하는 코드들이 있습니다.  
하지만 오류코드를 사용하면 오류코드를 반환하는 함수들을 사용할 때마다 if문으로 오류코드인지 체크해주어야 합니다.  
이런 오류 처리때문에 코드들의 가독성이 나빠지는 건 한순간입니다.  
  
<u>예외를 사용한다면 try-catch문으로 에러처리와 로직을 분기할 수 있어 훨씬 코드가 깔끔해집니다!</u>

## 🔥 함수를 짜는 방식  
  
처음부터 깔끔한 함수를 작성하는 것은 어려운 일입니다.  
  
처음에는 기능만 돌아가는 지저분하고 중복이 많은 함수를 제작합니다.  
그 다음에 단위테스트들을 만들고 코드정리를 합니다. (이때 수정될 때마다 계속 단위테스트들을 통과해야 합니다.)  
  
이렇게 계속 수정하다보면 깔끔한 함수를 얻을 수 있습니다.  
  
# 🚀 주석
  
> 주석은 내가 어떤 코드를 짰는지 설명, 메모하기 위해 필요한 기능입니다.  
  
주석은 잘쓰면 정말 유용한 기능이지만, 잘못쓰면 해로운 양날의 검입니다.  
좋은 주석은 무엇이고, 나쁜 주석은 무엇인지 어떻게 구분하는지 알면 클린코드에 한 걸음 가까워질 수 있습니다.  
  
🔔  
나쁜 코드를 주석으로 해결하려 하지마라.  
(브라이언 W. 커니핸, P.J. 플라우거) 
{: .notice}  
  
## 🔥 주석은 거짓말을 한다  
  
> 주석은 믿을 수 없습니다.  
  
```java
// HP를 회복하는 함수
public void Attack() {
  ...
}
```  
  
함수 이름은 Attack()인데 주석은 HP를 회복하는 함수라면 주석을 믿지 못하게 됩니다.  
프로그래머가 믿을 수 있는 것은 코드뿐입니다.  
  
<u>게다가, 주석은 코드의 변화에 따라가지 못하고 방치됩니다.</u>  
  
처음에는 A기능을 하는 함수에 A기능을 한다고 주석을 달아놓았을 때, 함수를 B기능으로 바꾸면 주석과 코드에 차이가 생겨버립니다.  
  
<u>그렇기에 주석은 유지보수에 시간이 더 할애되기 때문에 가급적 사용하지 않는 것이 좋습니다.</u>
  
## 🔥 주석은 나쁜 코드를 보완하지 못한다  
  
나쁜 코드에 주석을 단다고 이해가 쉬워지는 것은 아닙니다.  
클린 코드에서는 엉망진창으로 만든 코드를 주석으로 해결하려 하지말라고 경고합니다.  
  
그렇기에, 주석으로 코드에 대한 설명을 하는 것보다는 코드를 개선하는 방법을 찾는 것이 옳은 방향입니다.  
즉, 코드로 의도를 표현하는 것이 최고의 방법입니다.

```java
// 직원에게 복지 혜택을 받을 자격이 있는지 검사한다.
if ((employee.flags & HOURLY_FLAG) && (employee.age > 65))

if (employee.isEligibleForFullBenefits())
```  
  
위 코드보다 아래 코드가 훨씬 보기 좋습니다. 이렇게 주석이 아닌 코드로 의도를 표현할 수 있습니다.

## 🔥 좋은 주석  

결국 클린코드를 위해서는 주석을 가급적 사용하지 않는 방향이 좋다라고 결론이 나옵니다.  
그럼 좋은 주석은 무엇이고, 언제 사용해야 하는 걸까요?  
  
```java
// kk:mm:ss EEE, MMM dd, yyyy 형식
Pattern timeFormat = Pattern.compile("\\d*\\d:\\d* \\w*, \\w* \\d* \\d*");
```

이렇게 세부구현이 이해하기 어려울수밖에 없는 코드(Format등) 에 어떻게 되어있는지 정보를 제공해주는 주석은 좋은 주석입니다.  
  
```java
// 스레드를 많이 생성하여 시스템에 영향을 끼쳐 테스트를 만들도록 함
for (int i = 0; i < 25000; i++) {
  SomeThread someThread = ThreadBuilder.builder().build();
}

// 유저로부터 입력받은 값을 저장할 때 trim으로 공백제거 필요
String userName = userNameInput.trim();
```  
  
이렇게 왜 이 구문들을 작성하였는지 의도와 중요성을 설명하는 주석은 좋은 주석입니다.  
  
```java
// TODO: 경계값에 대한 테스트 코드 추가하기

// FIXME : 음수일 때 예외 던지기
```  
  
TODO와 FIXME 주석도 좋은 주석입니다.  
TODO 주석은 앞으로 할 일. 지금은 해결하지 않지만 나중에 해야할 일을 적어두는 주석입니다.
FIXME 주석은 문제가 있지만, 당장은 수정할 필요가 없는 코드에 적습니다. 가능하면 빨리 수정해야한다고 알려주는 주석입니다.  
  
두 주석다 IDE에서 하이라이팅되는 경우가 많으며, 검색이 쉬우므로 TODO를 검색하는 것만으로도 해야할 일이 무엇이 있는지 알수도 있습니다.  

## 🔥 주석보다는 어노테이션을 사용하자  

이건 어노테이션을 사용할 수 있는 언어들에 해당합니다 (Java...)   

함수나 객체위에 어노테이션을 붙이면 IDE에서 어노테이션을 붙인 함수나 객체에 설명이 추가됩니다.  
예를들어 @Deprecated (오래된 코드라는 경고) @NotThreadSafe (멀티쓰레드에서 안전하지 않다는 경고) 등이 있습니다.

```java
@Deprecated
class DeprecatedTestClass {

  @NotThreadSafe
  public void notThreadSafeTestFunc() {
  }

}
```
