# 🚀 Chapter 3-4 : 링크 에러

> 빌드에서 에러가 발생할 경우 컴파일 에러는 보통 문제를 파악하기 쉽지만, 링크 에러는 문제를 파악하기 어렵다.  
  
그렇기에 링크 에러의 원인에 대해 알아보아 더욱 원활하게 파악해볼 수 있도록 하자. 

## 🔥 확인할 수 없는 외부 참조

```c++
/////////////////////////////////////
// A.h

void FuncA();

/////////////////////////////////////
// Main.cpp

#include "A.h"

void main()
{
    FuncA();
}
```

예제의 결과는  

Main.obj: error LNK2019: "void _cdecl FuncA(void)" (?FuncA@@YAXXZ) 외부 기호  
(참조 위치: "void _cdecl main(void)" (?main@@YAXXZ) 함수)에서 확인하지 못했습니다.  
  
가 나오게 된다. 왜 이런 에러가 나왔을까?  
<u>링커는 FuncA의 본체를 찾기 위해서 컴파일된 목적 파일들을 모두 찾아보는데,  
전혀 찾을 수 없기 때문이다.</u>  
  
그러니, FuncA를 정의만 해주면 이 에러는 해결이 된다.  
    
보통 함수 본체가 정의된 소스 파일이 있는데도 이 에러가 발생하면  
그 소스 파일을 프로젝트에 추가 안했을 확률이 매우 높다.  

결과의 뜻은 Main.obj 는 Main.cpp의 컴파일된 목적 파일이며, 에러가 발생한 대상이다.  
FuncA는 정의를 찾을 수 없는 문제의 함수이며,  
FuncA가 main 함수 안에서 사용되었다는 의미를 가진다.  
  
> 이렇게 링크 에러 로그만 잘 파악해도 어디서 문제가 발생했는지 파악하기 쉽다.
  
## 🔥 여러 번 정의된 기호
  
```c++
/////////////////////////////////////
// A.h

void FuncA();

/////////////////////////////////////
// A.cpp

void FuncA()
{
}

/////////////////////////////////////
// B.cpp

void FuncA()
{
}

/////////////////////////////////////
// Main.cpp

#include "A.h"

void main()
{
    FuncA();
}
```

예제의 결과는  
  
B.obj: error LNK2005: "void _cdecl FuncA(void)" (?FuncA@@YAXXZ)이(가)  
A.obj에 이미 정의되어 있습니다.  
  
가 나오게 된다. 왜 이런 에러가 나왔을까?  
<u>A.cpp와 B.cpp 모두 함수 본체가 정의되어 있어 링커가 어느 함수 정의와 연결해야 하는지 기준이 모호해지기 때문이다.</u>  
  
보통 이런 에러는 같은 종류의 라이브러리들을 사용했거나, 라이브러리끼리 충돌했기 때문에 일어난다. 이런 경우에는  namespace를 사용하여 충돌이 발생하지 않도록 해결할 수 있다.  
  
결과의 뜻은 B.obj 는 B.cpp의 컴파일된 목적 파일이며, 에러가 발생한 대상이다.  
B.cpp의 FuncA라는 함수가 이미  
A.obj 즉 A.cpp에 이미 정의가 되어있다는 뜻이다.
