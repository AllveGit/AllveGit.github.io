# 🚀 Chapter 5-4 : 배열의 크기

## 🔥 배열의 크기

배열의 크기를 구할 때 사용하는 것이 sizeof 연산자이다.  
  
배열의 크기 = sizeof(배열 이름)  
배열의 요소 개수 = sizeof(배열 이름) / sizeof(요소 타입)  
  
으로 구할 수 있다.

<u>배열은 포인터로 변환하지 않는 이상 다차원이어도 sizeof로 크기를 정확하게 구할 수 있다.</u>  
배열을 포인터로 변환하면 sizeof의 인자로 전달하였을 때 배열의 크기를 정확하게 구할 수 없고,  
포인터의 크기인 4 or 8 바이트만 리턴이 된다.

## 🔥 배열의 요소 개수

> 배열 요소의 개수가 정해지는 시점은 배열이 생성되는 때가 아닌, 컴파일될 때이다.

<u>컴파일러가 컴파일 시점에 정해진 요소의 개수대로 배열을 생성하는 어셈블리를 작성하기 때문이다.</u>

즉, 실행 중에는 이미 정해진 크기의 배열 객체가 생성이 된다.  
  
```c++
void main()
{
  int count = 1024;
  int arr[count] = {0}; // Compile Error!
}
```  
  
이런 코드가 안되는 이유가 있었다.  
컴파일 시점에 배열의 요소 개수를 정해야 했으므로 실행중에 계속 바뀔수 있는 변수는  
배열 요소 개수를 정할 때 쓰일 수 없다.  
평범한 배열 요소 개수는 오직 상수로만 정할 수 있다.

상수 변수도 배열 요소 개수를 정할 때 쓰일 수 있었다.  
하지만, 상수 변수도 포인터를 이용하면 변수의 값을 바꿀 수 있는데?  
  
```c++
void main()
{
  const int count = 1024;

  int* pCount = (int*)&count;
  *pCount = 2048;

  int arr[count] = {0};
}
```

이렇게 하면, 상수 변수 1024의 값을 2048로 바꿀 수 있다. 컴파일도 성공한다!
하지만, 결과적으로 배열은 여전히 1024개의 요소 개수로 생성이 된다.  
왜냐하면, 컴파일러는 const가 바뀌지 않는 가정을 하고 있기 때문에 컴파일되는 시점에 const 객체들을 매크로처럼 취급을 한다.  
그렇기에, count는 1024로 취급을 하고 컴파일하는 것이다.  
  
<u>즉, 특수한 경우가 아닌 일반적인 경우에는 실행 시점에 배열 요소의 개수를 결정할 수 없다.</u>

## 🔥 가변 크기 배열 (VLA : Variable Length Array)

> 배열 요소의 개수가 실행 시점에 결정될 수 없는 건 단순히 예전 컴파일러들이 허용하지 않았기 때문이다.

new[] 를 통해 힙에 배열을 생성하면 가변 크기 배열이 가능하다고 생각할수도 있다.  
하지만, <u>new[]가 반환하는 것은 힙에 생성된 메모리 블록의 포인터이지, 배열자체가 아니다.</u>  
  
가변 크기 배열은 데이터와 힙 영역에서는 생성되는 것이 사실상 불가능하다.  
고로, <u>가변 크기 배열은 스택 영역에서만 생성될 수 있다.</u>  
데이터와 힙 영역도 생성할 순 있겠지만, 기존 프로그래밍 구조가 전면적으로 변경되어야 한다는 이유때문에 금지했다고 알면 될 것 같다.  
  
스택에 생성되는 배열은 실행중에 요소의 개수를 결정하는 것은 쉬운 일이다.  
왜냐하면, 배열을 생성하는 어셈블리에서 배열 메모리 할당을 위하여 스택을 늘리는 부분을   
스택 메모리를 할당하는 함수인 alloca로 대체하면 되기 때문이다.  
  
기술적으로 그리 큰 일이 아닌데, 컴파일러들은 왜 가변 크기 배열을 지원하지 않을까?  
왜냐하면 스레드 스택은 보통 1MB정도의 크기를 가지는데,  
<u>배열의 크기가 가변적이면 스택 오버플로우등 예외가 많이 생길수도 있기에 금지한다.</u>
