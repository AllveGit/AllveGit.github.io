# 🚀 Chapter 3-1 : 빌드의 구성

> 빌드는 최종 바이너리(실행 파일, 동적 라이브러리 등)을 만들기 위해  
거치는 과정을 모두 일컫는다.
  
빌드에는 크게 세 단계로 나눌 수 있다.
- 전처리
- 컴파일
- 링크

## 🔥 전처리

전처리는 컴파일 전에 실행되는 단계이다.  
  
예를 들어, 전처리 지시문인 #include가 없다면 어떻게 될까?  
include는 헤더 파일의 내용을 그대로 소스 파일에 붙이는 역할을 한다.  
그렇기에, 소스 파일마다 필요한 헤더 파일의 내용들을 그대로 붙여넣기를 했어야 할 것이다.  
  
이렇게 전처리문들이 없으면 개발자들이 비효율적인 고생들을 하게 된다.

> 즉 전처리란, 컴파일 이전에 비효율적인 작업들을 효율적이고 편하게 수행할 수 있도록 하며 컴파일 시점에 컴파일이 가능하도록 소스 파일을 원래 구문으로 복구시키는 과정을 의미한다.


## 🔥 컴파일

C++ 프로그래밍을 할 때 보통 파일 확장자는 .h, .cpp를 보게 된다.
  
<u>컴파일의 대상이 되는 파일은 바로 소스 파일(.cpp)</u>이다.  
**헤더 파일은 컴파일하지 않는다.** cpp 파일 안에 #include로 포함될 뿐이다. 
  
소스 파일이 컴파일되면 목적 파일(.obj)이 생성된다.  
소스 파일과 목적 파일은 **일대일 대응 관계이다.**  
  
소스 파일은 C++ 문법으로 이루어진 사람이 짠 코드이기에 <u>컴퓨터가 이해하지 못한다.</u>  
목적 파일은 이 소스 파일을 <u>기계가 읽을 수 있게 기계어로 변환</u>시킨 것이다.  
  
> 즉 컴파일이란, 사람이 이해할 수 있는 cpp 파일을 기계가 이해할 수 있는 기계어로 변환시키는 과정을 의미한다.

## 🔥 링크

최종 바이너리는 보통 여러 개의 소스 파일들로 구성되어 있다.
즉, 최종 바이너리는 여러 개의 목적 파일들이 구성 및 연결되어 있는 상태이다.
  
이때 <u>목적 파일들을 연결하는 과정을 링크</u>라고 한다.
  
예를 들어, 단순히 printf를 하는 코드가 있다고 해보자.  
  
```c++
#include <stdio.h>

int main()
{
    printf("Hello World!");
    return 0;
}
```
  
여기서 printf는 어디서 오는 것일까?  
printf.c 파일이 컴파일된 printf.obj에서 오는 것이다.  
  
그렇기에 printf가 정상 작동 되려면,  
printf를 호출하는 코드는 printf.obj가 프로세스 메모리에 올라간 주소로 변환하여 호출시켜줘야 한다.  
    
이 변환을 해주는 과정이 링크이다.  
  
> 즉 링크는, 많은 목적 파일들이 프로세스 메모리에 적재 되었을 때, 서로가 연결되어 정상작동할 수 있도록, 각각의 주소를 변경해주는 것을 의미한다.
