# 🚀 Chapter 5-5 : STL Vector

> 이전 파트에서 설명했던 가변 크기 배열은 거의 사용하지 않으며,  
사용할 일이 있어도 대신 STL Vector를 채용한다.  

STL Vector는 배열의 장점은 유지하면서 단점들은 덮을 수 있는 자료구조이다.

## 🔥 배열과 vector

실제 STL Vector의 사용방식을 보면 배열과 거의 일치하다.  
오히려, vector에 더 많은 기능과 장점이 있기에 상위호환으로 볼 수도 있다.  
즉, 배열 대신에 vector만 써도 충분할수도 있다.

하지만, <u>vector는 기능적으로 배열을 대체할 수 있겠지만, 완벽하게는 대체하지 못한다.</u>  

```c++
void Func(int (&arg)[3])
{
  // ...
}

void main()
{
  vector<int> v(3);
  Func(v); // Compile Error!
}
```

예제 코드에서는 <u>vector가 배열 타입으로 인자를 전달할 수 없다</u>는 것을 보여준다.  

이번에는, 배열 타입이 아닌 포인터 타입으로 전달해보겠다.

```c++
typedef int (*PARR)[3];

void Func(PARR pArg)
{
  cout << (*pArg)[0] << endl;
  cout << (*pArg)[1] << endl;
  cout << (*pArg)[2] << endl;
}

void main()
{
  vector<int> v(3);
  v[0] = 1; 
  v[1] = 2; 
  v[2] = 3;

  Func((PARR)&v);
}
```

예제 코드에서는 배열의 포인터 타입을 따로 정의하여 함수 인자로 전달받는다.  
vector를 정의한 포인터 타입으로 강제 변환해주어 전달한다.  

컴파일 에러는 나지 않겠지만 실행 결과는 이상하게 나올 것이다.  
결과 값 [ 1, 2, 3 ] 을 기대하고 함수를 만들었지만,  
쓰레기값 출력 혹은 메모리 접근 에러가 날 것이다.

이 이미지는 배열과 vector의 메모리 구조 차이를 보여준다.  
  
배열은 메모리 블록 자체를 나타내지만,  
vector는 _Myfirst 포인터가 실제 시작 메모리 주소를 가리키고 있기에  

<u>vector를 포인터로 변환</u>하여 첨자 연산자로 접근하게 되면  
<u>실제 값들이 있는 주소가 아닌, 클래스 객체 메모리 주소에 접근하여 이상한 값이 출력된다.</u> 

## 🔥 vector 인자 전달

 > 배열은 함수에 인자를 전달할 때 무조건 참조에 의한 호출(call by reference)로 실행된다.  

왜냐하면, 배열 이름 자체가 배열의 메모리 블록 주소를 의미하기 때문이다.  
그리고 다른 이유들도 있다.

일단 효율성 문제이다.  
만약 배열의 크기가 매우 크다면 <u>값에 의한 호출을 할 시에 인자 전달이 될 때마다,  
배열 요소들이 전부 복사가 이루어져야 할 것이다.</u> 그러므로 효율성이 떨어진다.
하지만, 참조에 의한 호출을 할 시에는 복사가 이루어질 필요가 없다.  
  
다음은 안정성 문제이다.  
만약 배열의 크기가 수MB 이상 크기라면 <u>값에 의한 호출을 할 시에 인자가 전달되면  
스택의 크기가 배열의 크기보다 작아 스택 오버플로우가 일어날 수 있다.</u>  
하지만, 참조에 의한 호출을 할 시에는 배열의 크기를 걱정할 필요가 없다.  
  
그렇다면, vector는 어떻게 인자로 전달이 될까?  
vector는 클래스이므로 클래스 객체의 인자 전달 방식을 따른다.  
  
값에 의한 호출로 전달할 경우, 복사 생성자가 호출되고 스택에 새로운 복사본이 생성된다.  
즉, <u>배열과는 다르게 값에 의한 호출이 된다는 것이다.</u>  
  
효율성 문제는 그대로 있지만, 안정성 문제는 괜찮다.  
왜냐하면 vector의 요소들은 힙 영역에 있고, 복사될 때도 힙 영역에 복사가 된다.  
그러므로, 스택에 vector 요소들이 저장되지 않으므로 스택 오버플로우가 일어나지 않는다.  
  
🔔  
vector를 함수에 인자 전달로 사용할 때에는 가능하면 참조에 의한 호출로 전달해주자.  
효율성이 떨어질 수 있기 때문이다.
