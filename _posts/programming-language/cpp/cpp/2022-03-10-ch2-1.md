# 🚀 Chapter 2-1 : 선언과 정의의 구분

## 🔥 정의

> C++에서의 정의는 객체를 실제로 만들겠다는 것을 의미한다.
즉, 만들 객체를 위한 메모리 영역을 확보하는 것을 말한다.

```c++
class Test
{
    
};

int a; //(1)

int main()
{
    int b; //(2)
    Test c; //(3)
}

int Add(int InA, int InB) //(4)
{
    return InA + InB;
}
```
  
<br>
(1)~(4)는 모두 메모리 영역을 확보하기 위해 정의를 한 구문들이다.
  
(4)번 케이스, <u>함수도 메모리 영역을 차지한다.</u>     
함수 본체가 있을 경우, **함수 본체를 어셈블리로 변환하고**, 그 어셈블리가 코드 영역에 저장이 된다.  
따라서, 함수도 일반 객체처럼 메모리 공간을 차지한다.
  
하지만, 클래스의 경우 정의 개념이 달라진다.  
클래스에서의 정의는 <u>클래스가 어떻게 구성되어 있는지 알 수 있는 설계도</u>를 의미한다.

기존 다른 정의들과 다른 점은, 클래스 설계도는 프로그램이 사용하는 메모리가 아닌, <u>컴파일러가 사용하는 메모리</u>에 저장이 된다.  
다만, 클래스 멤버 함수는 기존 함수와 같이 어셈블리로 변환해 코드 영역의 메모리에 저장된다.

🔔  
정의란 어떤 대상이 실제로 존재할 수 있도록 메모리 영역을 만들어주는 작업이다.  
클래스 정의는 클래스가 어떻게 구성되어 있는지를 설계도를 만드는 것을 의미한다.
{: .notice}
  
## 🔥 선언

> 선언이란 실제 객체가 존재하고 있음을 알리는 것을 의미한다.
  
그런데 정의도 객체를 정의하는 순간 해당 객체가 알려지게 된다.  
선언과 정의는 둘이 다른 것이 아닌 정의가 선언을 포함한다. 그렇기에 <u>정의를 하면 선언도 된다는 것이다.</u>

그런데도 선언이 굳이 있는 이유는 <u>선언이 필요할 때가 있기 때문</u> 이다.

```c++
/////////////////////////////////////
// A.cpp

int g_Val;
int Func()
{
    return 1;
}

/////////////////////////////////////
// Main.cpp

void main()
{
    g_Val = Func(); // Compile Error!
}
```

예제는 A.cpp에서 전역으로 g_Val과 Func를 정의 및 선언하였다.  
Compile Error가 나는 이유는 이 **선언 범위가 A.cpp에만 해당**된다는 것이다.

그렇기에 g_Val과 Func라는 전역함수가 있다는 <u>선언</u>을 해주어야 Main.cpp에서도 존재를 알 수 있다.  

```c++
/////////////////////////////////////
// A.cpp

int g_Val;
int Func()
{
    return 1;
}

/////////////////////////////////////
// Main.cpp

extern int g_Val;   // (1)
int Func();         // (2)

void main()
{
    g_Val = Func();
}
```

(1), (2) 같이 각각 객체가 존재한다는 선언을 해주면 컴파일 에러가 없어지게 된다.
  
❗  
정의는 선언을 포함하므로 Main.cpp에서도 정의를 해주면 되지 않을까?  
그렇게 되면 링크 에러가 발생한다.  
둘 중 어느 것을 선택해야 하는 지 모호해지기 때문에 중복 정의는 허용되지 않는다.
{: .notice--warning}
  
## 🔥 extern & static

> static은 영역 내부에서만 사용할 수 있게 하는 것을 의미한다.
extern은 영역 외/내부 모두 사용할 수 있게 하는 것을 의미한다.

```c++
/////////////////////////////////////
// A.cpp

extern int g_Val = 0;
static int s_Val = 0;

/////////////////////////////////////
// Main.cpp

extern int g_Val;
extern int s_Val;

void main()
{
    g_Val = 1;
    s_Val = 2; // Link Error
}
```
  
Link Error가 나는 이유는 <u>static으로 정의된 변수는 다른 소스파일 (cpp) 에서는 링크되지 못하기 때문</u>이다.

위 예제에서 extern이 두번 쓰였는데 어느 것이 정의이고, 어느 것이 선언인지 모호하다. 그래서 extern에는 규칙이 있다.  
- extern을 생략하거나, 값을 초기화한다면 정의라고 판단  
- extern type name; 형식으로만 사용되면 선언이라고 판단

```c++
int g_Val1;             // extern을 생략했기에 "정의"
extern int g_Val2 = 1;  // 값을 초기화 했기에 "정의"
extern int g_Val1;      // extern type name; 형식으로만 사용되었기에 "선언"
```

🔔   
extern은 다른 소스파일에서도 사용 가능하도록 링크를 허용 하는 것  
static은 다른 소스파일에서는 사용 못하도록 링크를 허용하지 않는 것 이라고 이해하면 된다.
{: .notice}
  
## 🔥 클래스 선언과 정의

일반적으로 객체나 함수는 가상 메모리 영역을 차지하지만, 클래스는 오직 개념적인 구조로만 존재한다.  
<u>클래스는 메모리 영역과는 전혀 상관이 없다는 의미</u>이다.

클래스는 사용자가 직접 만드는 타입이라봐도 무방하다.  
그렇다면 다른 소스파일에 정의되어 있는 클래스를 선언하면 그대로 쓸 수 있을까?

**클래스 전방선언만 한채로 클래스를 사용하게 되면 Compile Error가 날것이다.**  
왜냐하면 클래스 전방선언은 그 타입이 클래스라는 것만 알려주고 내용물은 알려주지 않기 때문에  
얼마만큼의 메모리 영역을 차지해야 하는지 컴파일러는 모른다.

클래스의 정의는 오직 정의된 소스파일에서만 알 수 있다. <u>그렇기에 컴파일러에게 필요한 것은 클래스의 선언이 아닌 클래스의 정의이다.</u>  
사용할 소스파일에서도 똑같이 클래스를 정의해주면 에러가 없어지게 된다. 이것이 헤더파일에 클래스의 정의를 넣는 이유이다.

**클래스의 정의의 중복 허용으로 인해 생기는 희귀한 버그들도 있다.**

```c++
/////////////////////////////////////
// A.cpp

class Test
{
public:
    Test(int InA, int InB)
    {
        m_ValA = InA;
        m_ValB = InB;
    }

    int m_ValB;
    int m_ValA;
};

extern Test g_Test;

int GetValA()
{
    return g_Test.m_ValA;
}

/////////////////////////////////////
// Main.cpp

class Test
{
public:
    Test(int InA, int InB)
    {
        m_ValA = InA;
        m_ValB = InB;
    }

    int m_ValA;
    int m_ValB;
};

Test g_Test(1, 2);

int GetValA();

void main()
{
    cout << GetValA();
}
```

이 값의 결과는 Main.cpp에서 m_ValA에 1을 집어넣었지만, 2가 출력되게 된다.

예제에서의 중요한 점은 **클래스 멤버변수 정의의 순서가 다르다는 점** 이다.

이런 결과가 일어난 이유는 <u>클래스가 멤버변수에 접근할 때에는 클래스 정의에 의한 메모리 오프셋을 기준으로 접근</u>을 하게 된다.  
멤버변수 정의의 순서대로 메모리 오프셋이 정해지기 때문에,

- Main.cpp의 Test에서 m_ValA의 오프셋은 0, m_ValB의 오프셋은 4
- A.cpp의 Test에서 m_ValA의 오프셋은 4, m_ ValB의 오프셋은 0
  
그렇기에 A.cpp의 m_ValA는 B.cpp의 m_ValB와 같기에 결과가 2가 출력이 된다.
